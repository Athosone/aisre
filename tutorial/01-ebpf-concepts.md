# Chapter 1: eBPF Concepts You Need

This chapter gives you the mental models for eBPF that you'll use in every subsequent chapter. No code to write — just read and understand.

## What is eBPF?

eBPF lets you run small, sandboxed programs inside the Linux kernel without modifying kernel source or loading kernel modules. Think of it as "user-defined callbacks that the kernel runs at specific hook points."

The key constraint: eBPF programs are **verified** before they run. The kernel's BPF verifier statically analyzes your program to guarantee it:
- Always terminates (no unbounded loops)
- Never accesses invalid memory
- Uses only allowed helper functions
- Stays within the instruction limit (~1M instructions)

This means the verifier will reject code that a normal C compiler would happily compile. You'll fight the verifier — that's normal.

## The Pieces

### 1. Programs

An eBPF program is a C function compiled to BPF bytecode. It attaches to a **hook point** in the kernel. When that hook fires, the kernel runs your function.

Hook points we use:
- **Tracepoints** (`SEC("tracepoint/syscalls/sys_enter_connect")`) — fire on kernel events like syscall entry/exit. Stable ABI, safe to use.
- **`inet_sock_set_state`** — a tracepoint that fires when a TCP socket changes state (e.g., becomes ESTABLISHED or CLOSE).

Each program is a separate `SEC("...")` function in a `.c` file.

### 2. Maps

Maps are key-value stores shared between eBPF programs and userspace. They survive across program invocations — this is how you maintain state.

Map types we use:

| Type | Purpose | Example |
|------|---------|---------|
| `BPF_MAP_TYPE_HASH` | General key-value lookup | `conn_info_map`: connection tracking |
| `BPF_MAP_TYPE_PERCPU_HASH` | Per-CPU hash (no locking needed) | `active_syscall_map`: syscall scratch |
| `BPF_MAP_TYPE_RINGBUF` | Kernel→userspace event stream | `events`: HTTP events to Go |
| `BPF_MAP_TYPE_ARRAY` | Fixed-size indexed array | `config_map`: runtime config |

Maps are defined using BTF-style macros in `common.h`:

```c
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 65536);
    __type(key, struct conn_key);
    __type(value, struct conn_info);
} conn_info_map SEC(".maps");
```

### 3. Helpers

BPF helper functions are the kernel API available to eBPF programs. You can't call arbitrary kernel functions — only approved helpers.

Helpers we use:

| Helper | Purpose |
|--------|---------|
| `bpf_get_current_pid_tgid()` | Get current process+thread ID (upper 32 = tgid/pid, lower 32 = tid) |
| `bpf_ktime_get_ns()` | Monotonic nanosecond timestamp |
| `bpf_get_current_uid_gid()` | Get current UID/GID |
| `bpf_probe_read_user()` | Safely read userspace memory (e.g., syscall buffer) |
| `bpf_map_lookup_elem()` | Look up a value in a map |
| `bpf_map_update_elem()` | Insert/update a map entry |
| `bpf_map_delete_elem()` | Delete a map entry |
| `bpf_ringbuf_reserve()` | Reserve space in the ring buffer |
| `bpf_ringbuf_submit()` | Submit a reserved ring buffer entry |
| `bpf_ntohs()` | Network-to-host byte order (16-bit) |

### 4. Ring Buffer

The ring buffer (`BPF_MAP_TYPE_RINGBUF`) is how eBPF programs send events to userspace. The workflow:

```
Kernel side:                          Userspace side:
1. bpf_ringbuf_reserve(&events, sz)
2. Fill in the event struct
3. bpf_ringbuf_submit(event)    →     ringbuf.Reader.Read() returns the event
```

Ring buffers (since kernel 5.8) replaced perf event arrays. Advantages:
- Shared across all CPUs (no per-CPU waste)
- Preserves event ordering
- Supports variable-length records
- Lower overhead

## BPF CO-RE (Compile Once, Run Everywhere)

Our programs use CO-RE via `vmlinux.h` — a header generated from the running kernel's BTF (BPF Type Format) information. This gives us access to kernel struct definitions without depending on kernel headers.

```c
#include "vmlinux.h"  // All kernel types
#include <bpf/bpf_helpers.h>  // BPF helper function declarations
#include <bpf/bpf_core_read.h>  // CO-RE read macros
```

`vmlinux.h` is generated by: `bpftool btf dump file /sys/kernel/btf/vmlinux format c`

## Tracepoints and Syscall Arguments

When you attach to `tracepoint/syscalls/sys_enter_connect`, the kernel gives you a `struct trace_event_raw_sys_enter` with:
- `args[0]`, `args[1]`, ... — the syscall arguments (as `unsigned long`)

For `sys_exit_*`, you get `struct trace_event_raw_sys_exit` with:
- `ret` — the syscall return value

The mapping from `args[]` to actual parameters depends on the syscall. For example, `connect(int fd, struct sockaddr *addr, socklen_t addrlen)`:
- `args[0]` = fd
- `args[1]` = addr pointer
- `args[2]` = addrlen

You need to know the syscall signature to interpret the args correctly.

## The pid_tgid Trick

`bpf_get_current_pid_tgid()` returns a 64-bit value:
- Upper 32 bits: **tgid** (what userspace calls "PID" — the process ID)
- Lower 32 bits: **tid** (the thread ID within the process)

```c
__u64 pid_tgid = bpf_get_current_pid_tgid();
__u32 pid = pid_tgid >> 32;    // tgid = userspace PID
__u32 tid = (__u32)pid_tgid;   // tid = thread ID
```

This is important because a single thread can only be in one syscall at a time, so `pid_tgid` is a safe key for correlating `sys_enter_*` with `sys_exit_*`.

## The Verifier: Your Frenemy

The verifier will reject your code for reasons that seem unreasonable at first. Common patterns:

**Bounded access**: The verifier must prove at compile time that all memory accesses are in bounds.

```c
// REJECTED: verifier can't prove `len` is bounded
bpf_probe_read_user(buf, len, ptr);

// OK: verifier sees the static bound
if (len > MAX_PAYLOAD_CAPTURE)
    len = MAX_PAYLOAD_CAPTURE;
bpf_probe_read_user(buf, len & (MAX_PAYLOAD_CAPTURE - 1), ptr);
```

The `& (MAX_PAYLOAD_CAPTURE - 1)` mask is a common trick — it gives the verifier a provable upper bound even when `len` might have been modified between the `if` and the call.

**No unbounded loops**: Classic `for` loops are only allowed if the verifier can determine the iteration count. In practice, you unroll or use bounded iteration.

**Null checks after map lookups**: `bpf_map_lookup_elem()` can return NULL. You must check before dereferencing.

```c
struct conn_info *ci = bpf_map_lookup_elem(&conn_info_map, &key);
if (!ci)
    return 0;  // MUST check
ci->src_port = port;  // OK after check
```

**Stack size limit**: eBPF stack is 512 bytes. Large local arrays (like our `__u8 buf[512]`) consume the entire stack. If you need more, use per-CPU arrays or maps.
